## 4.2 技术实践过程

### 4.2.1 语音情感陪伴日记模块的实现

#### （一）端侧语音采集与 Push-to-Talk 交互

语音采集由 `AudioRecorder` 类负责，基于 Android 原生 `AudioRecord` API 实现，采样率固定为 **16000 Hz**、单声道、PCM 16-bit 格式，与 Whisper 模型的输入要求完全匹配。

```kotlin
// AudioRecorder.kt
const val SAMPLE_RATE = 16000
private const val CHANNEL_CONFIG = AudioFormat.CHANNEL_IN_MONO
private const val AUDIO_FORMAT = AudioFormat.ENCODING_PCM_16BIT
```

缓冲区大小取 `AudioRecord.getMinBufferSize()` 的 2 倍，在降低延迟的同时保证录音稳定性。录音线程（`readLoop`）持续读取音频帧，并实时计算 RMS 值用于 UI 波形显示：

```kotlin
private fun computeRms(buffer: ShortArray, count: Int): Float {
    var sum = 0.0
    for (i in 0 until count) {
        val s = buffer[i] / 32768.0
        sum += s * s
    }
    return sqrt(sum / count).toFloat()
}
```

系统内置静音检测机制：录音开始后 800ms 保护期内不触发，之后若 RMS 持续低于阈值 `0.01f`（约 -40dB）超过 `silenceTimeoutMs`（默认 1000ms），则自动触发 `onSilenceDetected` 回调，实现免手动松开的自动停录。

录音停止后，`stopRecording()` 将 `Short` 数组归一化为 `FloatArray`（除以 32768.0f），直接传入 Whisper 推理引擎。

#### （二）Whisper 端侧语音转写

项目通过 JNI 封装 `whisper.cpp` 的 Tiny 模型，在设备端完成离线语音转写，彻底避免语音数据上传云端。`WhisperProcessor` 以单例模式管理模型生命周期，在 `VoiceDiaryScreen` 启动时于后台线程异步初始化：

```kotlin

LaunchedEffect(Unit) {

        if (!whisperProcessor.isInitialized()) {
            val ok = whisperProcessor.initFromAssets(context)
        }
        withContext(Dispatchers.Main) {
            whisperReady = whisperProcessor.isInitialized()
                       && whisperProcessor.isNativeLibraryLoaded()
        }
    }
}
```

转写调用设置 30 秒超时保护，防止推理卡死：

```kotlin
val transcription = withTimeout(30_000) {
    whisperProcessor.transcribe(audioData)
}
```

对于录音时长不足 0.1 秒（少于 1600 个采样点）的情况，系统直接丢弃并提示用户重试，避免无效推理。

#### （三）情绪分析引擎的实现

情绪分析由 `EmotionAnalyzer` 对象（Kotlin `object` 单例）实现，采用**加权关键词匹配**策略，支持 8 种情绪类型：开心、满意、平静、担心、孤单、难过、生气、不适。

每种情绪对应一个关键词集合，例如：

```kotlin
private val lonelyKeywords = setOf(
    "孤单", "寂寞", "孤独", "一个人", "没人陪", "冷清", "想念",
    "思念", "想孩子", "想家", "没人", "独自", "空荡荡"
)
private val sickKeywords = setOf(
    "疼", "痛", "不舒服", "难受", "生病", "头晕", "乏力", "虚弱",
    "发烧", "咳嗽", "感冒", "肚子", "腰", "腿", "手", "脚",
    "胸闷", "心慌", "气短", "没力气", "不想动"
)
```

评分时，负面情绪（不适、孤单、难过、生气）权重系数为 3，担心为 2，正面情绪（开心）为 2，满意为 1，体现了对老年人负面情绪的优先感知：

```kotlin
scores[Emotion.SICK]      = countMatches(sickKeywords)    * 3
scores[Emotion.LONELY]    = countMatches(lonelyKeywords)  * 3
scores[Emotion.SAD]       = countMatches(sadKeywords)     * 3
scores[Emotion.ANGRY]     = countMatches(angryKeywords)   * 3
scores[Emotion.WORRIED]   = countMatches(worriedKeywords) * 2
scores[Emotion.HAPPY]     = countMatches(happyKeywords)   * 2
scores[Emotion.SATISFIED] = countMatches(satisfiedKeywords) * 1
```

最终情绪强度在基础强度值上叠加匹配分数，并限制在 0-100 范围内：

```kotlin
val intensity = (topEmotion.key.intensity + topEmotion.value * 5).coerceIn(0, 100)
```

`emotionDistributionJson()` 方法将情绪列表序列化为 JSON 字符串（如 `{"开心":3,"平静":2,"孤单":1}`），存入 `EmotionLogEntity.emotionDistributionJson` 字段，供子女端周报可视化使用。

#### （四）多轮对话会话管理

`ConversationManager` 采用单例模式，以 `Dispatchers.IO` 协程上下文管理所有数据库操作，保证主线程不阻塞。

**会话生命周期**分为三个阶段：

1. **startSession()**：创建 `ConversationSessionEntity` 并写入数据库，返回 `sessionId`；
2. **saveUserMessage() / saveAssistantMessage()**：每轮对话分别保存用户消息和 AI 回复，用户消息在保存时同步调用 `EmotionAnalyzer.analyze()` 完成情绪标注；
3. **endSession()**：汇总本次会话所有用户消息的情绪，计算主导情绪和平均强度，生成会话摘要，并触发 `updateTodayEmotionLog()` 更新当日情绪日志。

```kotlin
// ConversationManager.kt - endSession 核心逻辑
val emotions = userMessages.map { it.emotion }.filter { it.isNotBlank() }
val dominant = EmotionAnalyzer.dominantEmotion(emotions)
val avgIntensity = userMessages.map { it.emotionIntensity }.average().toInt()
val summary = "本次对话${userMessages.size}轮，主要情绪：$dominant。内容摘要：$contentSummary"
```

**多轮上下文回复**：`generateReply()` 从数据库取最近 6 条历史消息作为上下文，优先调用云端 LLM（通义千问），失败时降级为本地规则回复：

```kotlin
val history = db.conversationMessageDao().getBySessionSync(sessionId).takeLast(6)
val reply = try {
    llmService.generateConversationReply(
        userMessage, emotion, history, userName
    ) ?: generateLocalReply(userMessage, emotion, userName)
} catch (e: Exception) {
    generateLocalReply(userMessage, emotion, userName)
}
```

本地降级回复按情绪类型给出固定模板，确保在无网络环境下系统仍可正常陪伴：

```kotlin
private fun generateLocalReply(content: String, emotion: String, userName: String?): String {
    return when (emotion) {
        "孤单" -> "我一直在这里陪着${name}，有空也可以给孩子们打个电话聊聊。"
        "不适" -> "身体不舒服要注意休息，如果一直不好记得告诉家人或去看医生哦。"
        // ...
    }
}
```

#### （二）情绪日志的自动汇总机制

每次会话结束时，`updateTodayEmotionLog()` 自动汇总当天所有会话数据，写入或更新 `emotion_log` 表中当天的记录（通过 `OnConflictStrategy.REPLACE` 实现幂等写入）：

```kotlin
private suspend fun updateTodayEmotionLog() {
    val todayStart = getTodayStartTimestamp()  // 当天零点时间戳
    val todayEnd = todayStart + 24 * 60 * 60 * 1000L - 1

    val sessions = db.conversationSessionDao().getByTimeRange(todayStart, todayEnd)
    val messages = db.conversationMessageDao().getByTimeRange(todayStart, todayEnd)
    val userMessages = messages.filter { it.role == "user" }
    val emotions = userMessages.map { it.emotion }.filter { it.isNotBlank() }

    val dominant = EmotionAnalyzer.dominantEmotion(emotions)
    val distJson = EmotionAnalyzer.emotionDistributionJson(emotions)

    val existing = db.emotionLogDao().getByDay(todayStart)
    val log = EmotionLogEntity(
        id = existing?.id ?: 0,
        dayTimestamp = todayStart,
        dominantEmotion = dominant,
        emotionDistributionJson = distJson,
        conversationCount = sessions.size,
        totalMessages = messages.size,
        summary = "今日对话${sessions.size}次，主要情绪：$dominant",
        sentToFamily = existing?.sentToFamily ?: false,
        createdAt = existing?.createdAt ?: System.currentTimeMillis()
    )
    db.emotionLogDao().insert(log)
}
```

`getTodayStartTimestamp()` 通过 `Calendar` 将时、分、秒、毫秒全部置零，确保同一天的多次会话始终汇总到同一条日志记录。

    // ...

## 5.1 语音识别模块验证

### 5.1.1 Whisper 端侧推理可用性验证

在华为 Nova 系列真机上对 Whisper Tiny 端侧模型进行了实际测试。测试过程中发现，Android 原生 `SpeechRecognizer` 在华为设备上因 HiTouch 系统服务长期占用 `VOICE_RECOGNITION` 音频源，导致 `SpeechRecognizer` 持续返回 `ERROR_RECOGNIZER_BUSY`（错误码 8），无法正常工作。

基于此，项目完全放弃 Android 原生语音识别，改为纯 Whisper 本地推理方案。实测结果如下：

**表5.1 Whisper Tiny 端侧推理性能测试结果**

| 测试项 | 测试结果 |
|--------|----------|
| 模型初始化成功率 | 100%（冷启动约 2-3 秒） |
| 短句识别速度（5秒音频） | 约 14 秒（realtime factor ≈ 0.36x） |
| 普通话识别准确率（安静环境） | 可用，常见词汇识别正确 |
| 离线可用性 | 完全离线，无需网络 |
| 设备兼容性 | 华为 Nova 系列（arm64-v8a）验证通过 |

注：realtime factor 0.36x 表示处理 1 秒音频需约 2.8 秒，对于老年人短句输入（通常 3-8 秒）总延迟在可接受范围内。

## 5.2 情绪分析模块验证
