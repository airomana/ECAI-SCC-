# 银发智慧膳食管家（ElderCare AI）技术研究报告

---

## 1 问题聚焦

### 1.1 研究背景

第七次全国人口普查数据显示，中国60岁及以上人口已达2.64亿，占总人口18.7%，预计2030年突破25%。在深度老龄化背景下，老年群体在饮食健康管理中面临三类核心痛点：

1. **认知障碍**：新式菜名与营养术语理解困难，如"舒芙蕾""轻食沙拉"等表达方式令老年人无所适从；
2. **记忆缺失**：慢病饮食禁忌易忘记，营养摄入状况难追踪，健康风险难控制；
3. **情感缺失**：独居用餐缺乏陪伴，专业指导获取困难，心理健康受影响。

对30位60岁以上老人的深度访谈揭示：86.7%的受访者表示"看不懂菜名"；73.3%的慢病患者"经常忘记哪些食物不能吃"；80%的独居老人希望"有人陪着说说话"。

现有软件应用普遍存在落地困境：识别结果缺少通俗解释、弱光/反光/抖动场景识别不稳定、难以结合慢病档案提供个性化风险提示、交互复杂适老化不足。

### 1.2 研究目标

本项目提出面向银发群体的智能膳食管理方案——银发智慧膳食管家（ElderCare AI），构建四大功能模块形成闭环：

1. 智能菜单识别与大白话翻译；
2. 冰箱智能管理与保质期预警；
3. 语音情感陪伴日记；
4. 子女远程守护中心。

核心目标：降低饮食信息理解门槛、降低健康风险、降低子女远程照护成本。

## 2 相关工作

### 2.1 端侧语音识别技术

OpenAI 于2022年发布的 Whisper 模型采用 Transformer 编码器-解码器架构，在68万小时多语言语音数据上训练，具备强大的多语言识别能力。Whisper.cpp 是其 C++ 移植版本，支持通过 JNI 在 Android 设备上进行端侧推理，Tiny 模型参数量约39M，在中端 ARM 设备上可实现实时因子约0.3-0.5x的推理速度，适合老年人短句语音输入场景。

相比 Android 原生 `SpeechRecognizer`，Whisper 端侧推理的优势在于：完全离线、无需网络、不依赖厂商系统服务（规避了华为 HiTouch 等系统服务占用 `VOICE_RECOGNITION` 音频源导致的 `ERROR_RECOGNIZER_BUSY` 问题）。

### 2.2 端侧 OCR 技术

PaddleOCR 是百度飞桨开源的端到端 OCR 系统，采用 DB（Differentiable Binarization）文本检测 + CRNN 文本识别的两阶段架构。通过 FastDeploy 框架量化压缩后，可在移动端实现轻量化部署，支持 Lite FP16 精度推理，在中端 Android 设备上识别速度可达实用水平。

### 2.3 大语言模型与视觉模型

阿里云通义千问（Qwen）系列模型提供文本生成（qwen-turbo）和视觉理解（qwen-vl-max）能力，通过 DashScope API 对外提供服务。qwen-vl-max 支持图像输入，可对冰箱食材图像进行识别和新鲜度判断；qwen-turbo 支持多轮对话，适合老年人陪伴场景的情感化回复生成。

### 2.4 检索增强生成（RAG）

RAG（Retrieval-Augmented Generation）技术将外部知识库检索与大语言模型生成相结合，可有效减少模型幻觉、提升专业领域回答的准确性。在饮食健康建议场景中，将权威营养学知识（如中国营养学会《食物成分表》）构建为本地知识库，在 LLM 生成建议前先检索相关条目注入 Prompt，可显著提升建议的可靠性和保守性。

### 2.5 适老化 UI 设计

工信部《互联网应用适老化及无障碍改造专项行动方案》明确要求主流 APP 完成适老化改造。适老化 UI 的核心原则包括：大字号（≥18sp）、高对比度（对比度比≥4.5:1）、超大触控目标（≥48dp）、减少操作步骤、提供语音反馈。

#### 4.2.1 图像采集、OCR 识别与菜名结构化提取

**图像采集**：`MenuScanScreen` 支持拍照和相册两种输入方式。拍照通过 `ActivityResultContracts.TakePicture` 调起系统相机，使用 `FileProvider` 将照片写入应用缓存目录（`menu_photo_${timestamp}.jpg`），避免 Android 7.0+ 的 URI 权限问题；相册选取在 Android 13+ 使用 `PickVisualMedia`（`READ_MEDIA_IMAGES` 权限），Android 12 及以下使用 `GetContent`（`READ_EXTERNAL_STORAGE` 权限），两套路径统一收敛到 `capturedBitmap`。图像获取后进入 `CropImageDialog` 裁剪对话框，用户可拖拽四角调整识别区域，有效减少无关背景干扰。

**图像预处理**：`processImage()` 在 OCR 前调用 `ImageProcessor.assessImageQuality()` 评估图像质量（0-100分）。质量分低于 70 时自动执行亮度调整（+10）和对比度增强（×1.1）；无论质量如何，均将图像缩放至最大 1920×1920px，在保证 OCR 精度的同时降低推理内存占用：

```kotlin
var processedBitmap = bitmap
if (quality.score < 70) {
    var adjusted = ImageProcessor.adjustBrightness(processedBitmap, 10)
    adjusted = ImageProcessor.adjustContrast(adjusted, 1.1f)
    processedBitmap = ImageProcessor.scaleBitmap(adjusted, 1920, 1920)
} else {
    processedBitmap = ImageProcessor.scaleBitmap(processedBitmap, 1920, 1920)
}
```

**端侧 OCR 识别**：菜单识别采用 PaddleOCR（FastDeploy 移动端版本），通过 `PaddleOcrProcessor` 类封装，完全在设备端完成文字提取，无需上传图片至云端。初始化时从 `assets/paddle_ocr/` 目录加载检测（det）、识别（rec）、方向分类（cls）三个子模型，均采用 Lite FP16 精度推理：

```kotlin
val config = OcrConfig().apply {
    detModelFileName = "det"; recModelFileName = "rec"; clsModelFileName = "cls"
    recRunPrecision = RunPrecision.LiteFp16
    detRunPrecision = RunPrecision.LiteFp16
    clsRunPrecision = RunPrecision.LiteFp16
}
```

`recognize()` 在 `Dispatchers.IO` 线程执行，首次调用时自动触发 `init()`，后续调用复用已加载模型，识别结果通过 `result.simpleText.trim()` 获取拼接后的纯文本。`close()` 在 `DisposableEffect` 的 `onDispose` 中调用，确保页面退出时释放模型内存。

**菜名结构化提取**：`DishNameExtractor` 对 OCR 输出的原始文本进行结构化解析，处理流程分为九步：预处理（统一换行符）→ 移除价格信息（正则匹配 `¥12.5`、`38元` 等格式）→ 移除编号序号（`1.`、`①`、`A.` 等）→ 移除单位信息（`2份`、`500g` 等）→ 移除后缀标签（`推荐`、`热销`、`售罄` 等）→ 移除括号内容 → 清理特殊字符 → 有效性验证（长度 2-30 字符、至少含 1 个中文字符）→ 置信度计算。

置信度综合长度（2-6字最优）、菜名关键词命中数（食材词、烹饪方式词）、原始行是否含价格/编号等因素计算 0-1 分值：

```kotlin
private fun calculateConfidence(name: String, originalLine: String): Float {
    var confidence = 0.5f
    when (name.length) { in 2..6 -> confidence += 0.25f; in 7..10 -> confidence += 0.2f; else -> confidence += 0.1f }
    val keywordMatchCount = dishKeywords.count { name.contains(it) }
    confidence += when (keywordMatchCount) { 0 -> 0f; 1 -> 0.15f; 2 -> 0.25f; else -> 0.3f }
    if (Regex("""[¥￥$]\d+|\d+元""").containsMatchIn(originalLine)) confidence += 0.2f
    return confidence.coerceIn(0f, 1f)
}
```

最终结果按置信度降序排列，去重后最多返回 30 个候选，后续处理取前 10 个（`limitedCandidates = dishCandidates.take(10)`）以控制 API 调用次数。

#### 4.2.2 三级知识库匹配与健康风险分级评估

**三级知识库匹配**：对每个菜名候选，`processImage()` 依次尝试三级匹配，优先使用本地知识库数据：

```kotlin
val dishInfo = db.dishDao().getByName(candidate.name)          // 1. 精确匹配
    ?: db.dishDao().searchByName(candidate.name, limit = 5)    // 2. SQL LIKE 模糊搜索
        .firstOrNull()
    ?: run {                                                    // 3. 智能相似度匹配
        val allDishes = db.dishDao().getAllOnce()
        DishKnowledgeMatcher.matchDish(candidate.name, allDishes)
    }
```

`DishKnowledgeMatcher.matchDish()` 实现了三层智能匹配：精确匹配（规范化后完全相等）→ 包含匹配（处理"红烧猪肉"匹配"红烧肉"的情况）→ Jaccard 相似度匹配（字符集合交集/并集比值，阈值 > 0.5 才采用）。知识库命中时，`DishResult` 的 `dishInfo` 字段携带菜品的主要成分、烹饪方式、营养信息、标签、不适合人群等结构化数据。

**健康风险分级评估**：`HealthRiskEvaluator` 实现了五步分层评估架构：

**第一步：烹饪工艺分级评估**（按优先级匹配，只计算最高风险等级）

| 风险等级 | 关键词示例 | 分值 |
|---------|-----------|------|
| 极高风险 | 炸、油炸、深炸、复炸、干煸 | +6 |
| 高风险 | 红烧、糖醋、蜜汁、拔丝、烧烤 | +4 |
| 中风险 | 卤、酱、腌、煎、炒、烧 | +2 |
| 低风险（减分） | 清蒸、白灼、水煮、凉拌、清炒 | -2 |

**第二步：食材分类分级评估**（可累加多项）：高脂动物（肥肉/五花肉/猪蹄，+5）、加工肉类（腊肉/香肠/火腿，+4）、高嘌呤/内脏（肝/肾/鸭血，+4）、精制碳水（糯米/年糕/汤圆，+2）。

**第三步：组合效应评分**：油+糖叠加（+3）、盐+脂叠加（+2）、淀粉+油脂叠加（+2），捕捉单一维度无法识别的复合风险。

**第四步：个性化健康档案深度耦合**：结合用户慢病信息进行个性化加分，覆盖糖尿病（升糖指数关键词，+8）、高血压（高钠关键词，+6）、高血脂/冠心病（饱和脂肪关键词，+8）、痛风（高嘌呤关键词，+8）、慢性肾病（高蛋白/高盐关键词，+6）五类慢病：

```kotlin
disease.contains("糖尿病") || disease.contains("血糖") -> {
    val diabetesKeywords = listOf("甜","糖","蜜","淀粉","粥","勾芡","糯米","拔丝","糖醋","年糕","汤圆","粽子")
    diabetesKeywords.forEach { keyword ->
        if (dishName.contains(keyword)) { score += 8; reasons.add("⚠️ 糖尿病患者：此菜品升糖指数高，应严格控制"); return@forEach }
    }
}
```

**第五步：过敏原一票否决**：过敏原通过映射表扩展匹配（如"海鲜"→虾/蟹/鱼/贝/鱿鱼/章鱼等），命中时直接将风险分设为 99，输出 HIGH 等级，不受其他评分影响。最终风险等级：总分 ≥ 16 → HIGH（红色），8-15 → MEDIUM（黄色），< 8 → LOW（绿色）。

#### 4.2.3 RAG 知识库增强与本地降级建议生成

**RAG 知识库查询**：`DietKnowledgeRepository` 实现了基于本地 CSV 文件的 RAG 查询器。知识库文件位于 `assets/rag/diet_knowledge.csv`，字段格式为 `disease, keyword, risk_level, advice, source`，每行代表一条"疾病×关键词"的权威饮食建议。`loadFromCsv()` 在首次访问时懒加载（`by lazy`），通过 `BufferedReader` 逐行解析，支持双引号包裹含逗号的字段，跳过空行、表头和 `#` 注释行。

查询时遍历记录，疾病字段与用户健康档案慢病列表进行双向模糊匹配，关键词字段与菜品名称进行包含匹配，命中记录按风险等级（HIGH > MEDIUM > LOW）排序后取前 N 条：

```kotlin
val matched = records.filter { record ->
    val diseaseMatch = diseases.any { userDisease ->
        userDisease.contains(recordDiseaseLower) || recordDiseaseLower.contains(userDisease)
    }
    val keywordMatch = dishLower.contains(recordKeywordLower)
    diseaseMatch && keywordMatch
}
```

**RAG+LLM 建议生成**：检索到 RAG 记录时调用 `LlmService.generateDishAdviceWithRag()`，将权威知识注入 Prompt，LLM 被要求以权威知识为准、偏保守地生成大白话建议（temperature=0.3，max_tokens=400）；未检索到记录时降级调用 `generateDishAdviceFromProfileOnly()`，仅基于健康档案生成建议。LLM 的 System Prompt 明确约束了保守原则：

```kotlin
"你是一个非常保守的老年人健康饮食医生助理。
规则：
1. 如果权威知识提示'应避免''不推荐'，你必须严格遵守，不得放宽。
2. 如果你自己的知识和权威知识冲突，一律以权威知识和疾病限制为准。
3. 如果不确定安全性，也要偏保守，建议少吃或换成更清淡的菜。
4. 输出必须用简体中文、口语化、适合朗读，让老年人听得懂。
5. 输出长度控制在 40～80 字之间。"
```

**本地降级方案**：当 LLM 不可用或调用次数超限（`maxLlmCalls=10`）时，系统使用两套本地降级方案。`generateSimpleDescription()` 基于菜名关键词匹配生成大白话描述，覆盖烹饪方式（红烧/清蒸/糖醋/炸等）和主要食材（肉类/鱼/虾/豆腐/蔬菜等）两个维度，组合生成自然语言描述。`generatePersonalizedAdvice()` 结合知识库 `dishInfo` 和 `HealthRiskResult`，按过敏原警告 > 不适合人群警告 > 风险等级建议的优先级生成个性化建议：

```kotlin
return when {
    hasAllergy   -> "⚠️ ${userName}，这个${dishName}可能含有您过敏的成分，建议不要吃"
    notSuitable  -> "⚠️ ${userName}，这个${dishName}...，您有${disease}，建议少吃或不吃"
    riskLevel == HIGH   -> "${userName}，这个${dishName}...，建议少吃"
    riskLevel == MEDIUM -> "${userName}，这个${dishName}...，可以适量吃一点，但要注意控制分量"
    else         -> "✅ ${userName}，这个${dishName}...，比较适合您，可以放心吃"
}
```

最终建议优先使用 RAG+LLM 生成结果，LLM 不可用时回退到本地规则：`val finalAdvice = ragAdvice ?: basePersonalizedAdvice`。

#### 4.2.4 processImage 完整调用链

`processImage(bitmap)` 是图像进入系统后的核心编排函数，在 `Dispatchers.IO` 协程中执行，完整调用链如下：

```
processImage(bitmap)
  ├─ 1. ImageProcessor.assessImageQuality()          → imageQuality（0-100分）
  ├─ 2. 图像预处理（亮度/对比度增强 + 缩放至1920×1920）
  ├─ 3. ocrProcessor.recognize(processedBitmap)      → recognizedText
  ├─ 4. DishNameExtractor.extractDishCandidates(text)→ dishCandidates（最多30个）
  ├─ 5. take(10)                                     → limitedCandidates
  └─ for each candidate:
       ├─ 6. db.dishDao() 三级匹配                   → dishInfo
       ├─ 7. riskEvaluator.evaluateRisk()             → riskResult（风险等级+原因列表）
       ├─ 8. generateSimpleDescription() / LLM        → plainDescription（大白话描述）
       ├─ 9. generatePersonalizedAdvice()              → basePersonalizedAdvice
       ├─ 10. dietKnowledgeRepository.query()         → ragRecords
       ├─ 11. LLM generateDishAdviceWithRag/FromProfileOnly → ragAdvice
       └─ 12. DishResult(name, description, finalAdvice, riskLevel, reasons, dishInfo)
  → scanResults（List<DishResult>）
  → TTS 播报摘要
```

单个菜品处理失败时通过 `try-catch` 跳过继续处理其他菜品，保证部分失败不影响整体结果。LLM 调用次数通过 `llmCallCount` 计数器限制在 `maxLlmCalls=10` 以内，超限后自动切换为本地降级方案，避免 API 费用失控。整个流程的关键设计原则是"保守优先"：RAG 知识库命中时 LLM 被约束为保守输出，本地降级时过敏原和慢病禁忌优先于通用建议，确保老年用户不会因系统错误而摄入有害食物。

#### 4.2.5 TTS 播报与结果展示

**TTS 播报策略**：识别完成后根据结果数量采用不同播报策略，单道菜详细播报菜名+大白话描述+健康建议，多道菜先摘要播报总数和高风险菜品数，再单独播报前 2 道高风险菜品的警告：

```kotlin
if (results.size == 1) {
    val message = "${userName}，识别到${result.name}。${result.description}。${result.healthAdvice}"
    ttsService.speak(message, priority = 1)
} else {
    ttsService.speak("${userName}，识别到${results.size}道菜，其中${highRiskCount}道菜需要特别注意", priority = 1)
    results.filter { it.riskLevel == RiskLevel.HIGH }.take(2).forEach { result ->
        ttsService.speak("${result.name}，${result.healthAdvice}", priority = 1)
    }
}
```

`priority = 1` 确保识别结果播报能插队到 TTS 队列头部，不被其他低优先级语音阻塞。OCR 识别到文字但未提取到菜名时，播报"已识别文字，但未提取到菜名，请查看识别结果"，引导用户查看原始 OCR 文本。

**DishResultCard 结果展示**：每道菜以卡片形式展示，背景色按风险等级区分（HIGH→`errorContainer` 红色，MEDIUM→`secondaryContainer` 黄色，LOW→浅绿色 `#E8F5E9`）。卡片默认折叠，点击展开后显示知识库详细信息（主要成分、烹饪方式、营养信息、标签）和详细风险原因列表。图像质量分低于 70 时，结果界面额外展示具体的改善建议（如"太暗""模糊"等），引导用户重拍以获得更准确的识别结果。

#### 4.3.1 图像采集、质量评估与预处理

**图像采集**：`FridgeScreen` 支持拍照和相册两种输入方式，分别通过 `ActivityResultContracts.TakePicture` 和 `PickVisualMedia`（Android 13+）/ `GetContent`（Android 12-）实现，相机拍照通过 `FileProvider` 创建临时 URI 避免路径暴露。图像获取后通过 `ContentResolver.openInputStream()` 解码为 `Bitmap` 传入 `FridgeViewModel.scanFridge()`。

**质量评估**：`FoodDetector.assessImageQuality()` 在调用云端模型前先对缩放至 256px 的小图进行四维本地评估，避免低质量图像浪费 API 调用：

| 检测维度 | 判定条件 | 提示信息 |
|---------|---------|---------|
| 亮度过低 | 均值 < 35 或暗像素占比 > 75% | 太暗了，请开灯或打开冰箱灯再拍 |
| 亮度过高/反光 | 均值 > 225 或亮像素占比 > 60% | 太亮或反光强，请换角度避开反光再拍 |
| 对比度不足 | 亮度标准差 < 18 | 画面对比太低，请靠近一点并对准冰箱内部 |
| 图像模糊 | 拉普拉斯方差 < 90 | 有点糊，请拿稳手机，靠近一点再拍 |

亮度采用 BT.709 标准（`L = 0.2126R + 0.7152G + 0.0722B`）计算，清晰度通过拉普拉斯算子方差（`-4×center + left + right + up + down`）衡量，方差越大边缘越清晰。

**图像预处理**：质量评估通过后，`preprocessBitmap()` 执行裁边（保留中心 90% 区域）和缩放（最长边≤1024px）两步处理，再以 JPEG 质量 75 压缩编码为 Base64 传入 API：

```kotlin
private fun preprocessBitmap(original: Bitmap): Bitmap {
    val cropped = cropBorder(original, 0.90f)
    return resizeToMaxDimension(cropped, 1024)
}
```

#### 4.3.2 视觉模型识别与结果解析

`LlmService.analyzeImage()` 将 Base64 图像与结构化 Prompt 组合，调用 qwen-vl-max 多模态模型。Prompt 遵循"安全优先"原则，要求模型严格输出 JSON 数组，每个食材对象包含 name、category、count、clarity、confidence、freshness、spoil_signs_observed 七个字段，并约束保守判断规则：只有 `spoil_signs_observed` 包含发霉/渗液/腐烂等明确迹象时才允许 `freshness=疑似变质`，看不清时 `confidence≤0.3`、`freshness=未知`。推理参数设置 `temperature=0.0`、`top_p=0.1` 确保输出确定性。

**结果解析与去重**：`FoodDetector` 对返回 JSON 进行字段白名单校验（`freshness`、`clarity` 不合法时重置为默认值）、类别归一化（如"水产"→"海鲜"），再以食材名归一化形式为 key 执行去重合并，合并策略遵循"取最差值"原则保障安全：

```kotlin
// 新鲜度取更差的（rank越高越差）
private fun worseFreshness(a: String?, b: String?): String {
    val rank = mapOf("新鲜" to 0, "一般" to 1, "快坏" to 2, "未知" to 3, "疑似变质" to 4)
    // 返回 rank 更高的那个
}
// 剩余天数取较小值；数量累加
private fun mergeDaysLeft(a: Int?, b: Int?): Int? = if (a == null) b else if (b == null) a else min(a, b)
```

`clarity=="看不清"`、`confidence<0.5f` 或 `freshness=="未知"` 的食材被标记为不确定，在 UI 中单独提示用户手动确认。

#### 4.3.3 RAG 规则引擎保质期推算

识别结果进入 `FridgeFoodRag` 进行保质期推算，这是一套基于本地 CSV 知识库的规则引擎，比简单的类别默认值更精细。

**知识库结构**：规则文件位于 `assets/fridge_food_data.csv`，每行代表一条食材规则，包含以下关键字段：

| 字段 | 说明 |
|------|------|
| food_name_std | 标准食材名 |
| aliases | 别名列表（`|` 分隔） |
| category | 食材类别 |
| keywords | 关键词（`|` 分隔） |
| storage | 存储方式（冷藏/冷冻/常温） |
| pack_state | 包装状态（未开封/开封/散装） |
| food_form | 食物形态（整个/切开/熟食） |
| shelf_life_min_days | 最短保质期（天） |
| shelf_life_max_days | 最长保质期（天） |
| bucket_candidates | 离散天数候选值（`|` 分隔） |
| default_bucket | 默认推荐天数 |
| risk_level | 风险等级（HIGH/MEDIUM/LOW） |
| spoil_signs | 变质迹象描述 |
| must_discard_if | 必须丢弃的条件 |
| advice_template | 建议文案模板 |
| safe_fallback_advice | 保守兜底建议 |

**规则匹配评分机制**：`findBestRule()` 对每条规则计算匹配分数，取最高分规则（分数需 ≥ 600 才认为有效匹配）：

```kotlin
// 评分规则（部分）
if (stdKey == nameKey) score += 1200          // 标准名完全匹配
if (aliasKeys.any { it == nameKey }) score += 1100  // 别名完全匹配
if (nameKey.contains(stdKey)) score += 700    // 标准名包含匹配
if (aliasKeys.any { nameKey.contains(it) }) score += 800  // 别名包含匹配
if (ruleCat == categoryKey) score += 150      // 类别匹配
if (rulePack == packKey) score += 80          // 包装状态匹配
if (ruleForm == formKey) score += 60          // 食物形态匹配
```

**食物形态推断**：`inferFoodForm()` 根据食材名称中的关键词推断形态，影响保质期计算：

- 含"熟食/剩菜/剩饭/炒/煮/炖"等词 → `COOKED`（熟食，保质期大幅缩短）
- 含"开封/拆封/已开"等词 → `OPENED`（已开封，保质期减半）
- 含"切开/切片/切块"等词 → `CUT`（已切开，保质期减半）
- 其他 → `WHOLE`（完整，按标准保质期）

**保质期天数离散化**：`chooseDaysLeft()` 将连续天数映射到 `bucket_candidates` 中的离散值（如 `1|2|3|5|7|14`），避免输出"还能放6.3天"这类不自然的表达，同时取模型估计值和规则值中的较小值，保持保守原则：

```kotlin
private fun chooseDaysLeft(modelDaysLeft: Int?, rule: Rule): Int? {
    val candidates = parseBucketCandidates(rule.bucketCandidates)
    val base = rule.defaultBucket?.let { bucketDownToCandidates(it, candidates) }
    val model = modelDaysLeft?.let { bucketDownToCandidates(it, candidates) }
    // 取较小值（更保守）
    val merged = when {
        base == null -> model
        model == null -> base
        else -> minOf(base, model)
    }
    return if (maxDays != null) merged?.let { minOf(it, maxDays) } else merged
}
```

**兜底规则**：当 RAG 规则库无匹配时，回退到 `ShelfLifeCalculator` 的内置规则表（覆盖50+种常见食材的精确保质期，以及按蔬菜子类别细分的默认值），再次回退才使用类别默认值。

#### 4.3.4 三色状态分级与适老化展示

`ShelfLifeCalculator.calculateFoodStatus()` 根据当前时间与过期时间的差值，将食材分为三个状态：

```kotlin
fun calculateFoodStatus(expiryTime: Long, currentTime: Long): FoodStatus {
    val remainingMillis = expiryTime - currentTime
    return when {
        remainingMillis <= 0L              -> FoodStatus.EXPIRED        // 已过期
        remainingMillis <= 2L * DAY_MILLIS -> FoodStatus.EXPIRING_SOON  // 2天内过期
        else                               -> FoodStatus.FRESH          // 新鲜
    }
}
```

### 4.4 语音情感陪伴日记模块

本模块是面向老年用户的核心交互入口，实现了"按住说话→端侧识别→情绪感知→AI回复→语音播报"的完整闭环。整体由 `VoiceDiaryScreen`（UI层）、`AudioRecorder`（采集层）、`WhisperProcessor`（识别层）、`EmotionAnalyzer`（分析层）、`ConversationManager`（对话层）、`TtsService`（播报层）六个组件协同完成。

#### 4.4.1 语音采集、Whisper 端侧推理与对话调用链

**语音采集**：`AudioRecorder` 基于 Android 原生 `AudioRecord` API 实现，采样率固定为 **16000 Hz**、单声道、PCM 16-bit 格式，与 Whisper 模型输入要求完全匹配。缓冲区大小取 `AudioRecord.getMinBufferSize()` 的 2 倍，录音线程 `readLoop` 每次读取半个缓冲区大小的数据以降低延迟。每帧音频数据同步计算 RMS 均方根值（归一化到 0-1），通过 `onRmsChanged` 回调驱动 UI 层的 `VolumeBar` 音量波形动画：

```kotlin
private fun computeRms(buffer: ShortArray, count: Int): Float {
    var sum = 0.0
    for (i in 0 until count) { val s = buffer[i] / 32768.0; sum += s * s }
    return sqrt(sum / count).toFloat()
}
```

**静音自动停录**：录音开始后 800ms 保护期（`SILENCE_GUARD_MS`）内不触发，之后若 RMS 持续低于阈值 `0.01f`（约 -40dB）超过 1000ms，自动触发 `onSilenceDetected` 回调。Push-to-Talk 模式下由用户松开按钮触发停录。录音停止后，`Short` 数组归一化为 `FloatArray`（除以 32768.0f）直接传入 Whisper。`forceStop()` 在每次 `startRecording()` 前调用，确保上一次录音资源完全释放。

**WhisperProcessor 模型生命周期**：`WhisperProcessor` 以双重检查锁定单例模式实现，确保全局只有一个模型实例。`initFromAssets()` 将 `assets/whisper/ggml-tiny-q8_0.bin` 复制到 `filesDir/whisper/`，若文件已存在则跳过复制直接加载，避免每次启动重复 IO。Native 库加载失败时通过 `nativeLibraryLoaded = false` 标志优雅降级，不抛出异常。三个 JNI 接口对应 whisper.cpp 的完整生命周期：

```kotlin
private external fun nativeInit(modelPath: String): Boolean
private external fun nativeTranscribe(audioData: FloatArray): String?
private external fun nativeRelease()
```

模型加载在 `LaunchedEffect(Unit)` 中于 `Dispatchers.IO` 线程异步执行，完成后切回主线程更新 `whisperReady` 状态，模型未就绪时录音按钮显示加载动画。转写调用设置 30 秒超时保护（`withTimeout(30_000)`），同时过滤不足 1600 个采样点（0.1 秒）的录音，避免无效推理。

**handleUserSpeech 对话调用链**：`handleUserSpeech(text)` 是语音识别结果进入对话系统的统一入口，完整调用链如下：

```kotlin
fun handleUserSpeech(text: String) {
    scope.launch {
        val sessionId = ensureSession()          // 1. 懒创建会话
        val userMsg = conversationManager.saveUserMessage(sessionId, text)  // 2. 保存消息+情绪分析
        val userName = db.userDao().getById(uid)?.nickname
        val reply = conversationManager.generateReply(                       // 3. 生成AI回复
            sessionId = sessionId, userMessage = text,
            emotion = userMsg.emotion, userName = userName
        )
        conversationManager.saveAssistantMessage(sessionId, reply)           // 4. 保存AI消息
        ttsService.speak(reply, priority = 1)                                // 5. TTS高优先级播报
    }
}
```

`ensureSession()` 实现懒创建逻辑：若 `currentSessionId` 为 null 则创建新会话，否则复用当前会话，保证同一次对话的所有消息归属同一 `sessionId`。

#### 4.4.2 情绪分析引擎与 TTS 语音播报

**情绪分析引擎**：`EmotionAnalyzer` 对象（Kotlin `object` 单例）采用加权关键词匹配策略，支持 8 种情绪类型，每种情绪对应一个枚举值，内置基础强度：

```kotlin
enum class Emotion(val label: String, val intensity: Int) {
    HAPPY("开心", 80), SATISFIED("满意", 70), CALM("平静", 50),
    WORRIED("担心", 40), SICK("不适", 35), LONELY("孤单", 30),
    SAD("难过", 25), ANGRY("生气", 20)
}
```

每种情绪对应一个关键词集合，以孤单类为例，覆盖了老年人常用的表达方式：`"孤单"、"寂寞"、"孤独"、"一个人"、"没人陪"、"冷清"、"想念"、"思念"、"想孩子"、"想家"、"没人"、"独自"、"空荡荡"`。不适类关键词（21个）覆盖了部位词（腰、腿、手、脚、肚子）和症状词（头晕、胸闷、心慌、气短），能有效捕捉老年人隐晦的健康信号。

**加权评分**：负面情绪（不适、孤单、难过、生气）权重系数为 3，担心为 2，正面情绪（开心）为 2，满意为 1，体现对老年人负面情绪的优先感知：

```kotlin
scores[Emotion.SICK]      = countMatches(sickKeywords)      * 3
scores[Emotion.LONELY]    = countMatches(lonelyKeywords)    * 3
scores[Emotion.SAD]       = countMatches(sadKeywords)       * 3
scores[Emotion.ANGRY]     = countMatches(angryKeywords)     * 3
scores[Emotion.WORRIED]   = countMatches(worriedKeywords)   * 2
scores[Emotion.HAPPY]     = countMatches(happyKeywords)     * 2
scores[Emotion.SATISFIED] = countMatches(satisfiedKeywords) * 1
```

最终情绪强度在枚举基础值上叠加匹配分数（×5），并限制在 0-100 范围内：`val intensity = (topEmotion.key.intensity + topEmotion.value * 5).coerceIn(0, 100)`。无任何关键词命中时返回默认情绪"平静"（强度 50），保证每条消息都有情绪标注。`emotionDistributionJson()` 将情绪列表统计为 JSON 字符串（如 `{"开心":3,"平静":2,"孤单":1}`），存入 `EmotionLogEntity.emotionDistributionJson` 字段，供子女端周报图表使用。`needsAlert()` 对孤单、难过、生气、不适四种情绪返回 `true`，供子女端关注提示逻辑使用。

**TtsService 语音播报**：`TtsService` 封装 Android `TextToSpeech`，以单例模式提供全局语音播报能力，支持队列管理和优先级插队。初始化时设置中文语言（`Locale.CHINESE`），若中文不支持则降级为英文，初始化完成后立即处理积压队列。

`speak(text, priority)` 中，`priority > 0` 的消息插入队列头部（插队），普通消息追加到队尾。AI 回复调用时传入 `priority = 1`，确保新回复能打断等待中的旧消息：

```kotlin
if (priority > 0) pendingQueue.add(0, text)  // 插队
else              pendingQueue.add(text)       // 排队
```

`UtteranceProgressListener` 监听每条语音的播报完成（`onDone`）和错误（`onError`）事件，两者均触发 `processQueue()` 继续处理下一条，保证队列不因异常中断。每次 `processQueue()` 调用 `TextToSpeech.QUEUE_FLUSH` 模式播报，确保新内容立即开始而不等待系统内部队列。

#### 4.4.3 多轮对话会话管理

`ConversationManager` 采用单例模式，以 `Dispatchers.IO` 协程上下文管理所有数据库操作，保证主线程不阻塞。

**会话生命周期**分为三个阶段：

1. **startSession()**：创建 `ConversationSessionEntity`（含 `startTime` 时间戳）写入数据库，返回自增 `sessionId`；
2. **saveUserMessage() / saveAssistantMessage()**：每轮对话分别保存消息，用户消息在保存时同步调用 `EmotionAnalyzer.analyze()` 完成情绪标注，结果存入 `emotion` 和 `emotionIntensity` 字段；
3. **endSession()**：汇总本次会话所有用户消息的情绪，计算主导情绪和平均强度，生成会话摘要，更新 `ConversationSessionEntity`，并触发 `updateTodayEmotionLog()` 更新当日情绪日志，同时写入 `DiaryEntryEntity` 兼容子女端展示：

```kotlin
suspend fun endSession(sessionId: Long) = withContext(Dispatchers.IO) {
    val userMessages = messages.filter { it.role == "user" }
    val dominant = EmotionAnalyzer.dominantEmotion(emotions)
    val avgIntensity = userMessages.map { it.emotionIntensity }.average().toInt()
    val summary = "本次对话${userMessages.size}轮，主要情绪：$dominant。内容摘要：$contentSummary"
    db.conversationSessionDao().update(session.copy(
        endTime = System.currentTimeMillis(),
        overallEmotion = dominant,
        emotionIntensity = avgIntensity,
        summary = summary,
        messageCount = messages.size
    ))
    updateTodayEmotionLog()
    // 写入 DiaryEntryEntity 兼容旧接口
    db.diaryEntryDao().insert(DiaryEntryEntity(...))
}
```

**多轮上下文回复**：`generateReply()` 从数据库取最近 6 条历史消息作为上下文，优先调用云端 LLM（通义千问），失败时降级为本地规则回复：

```kotlin
val history = db.conversationMessageDao().getBySessionSync(sessionId).takeLast(6)
val reply = try {
    llmService.generateConversationReply(userMessage, emotion, history, userName)
        ?: generateLocalReply(userMessage, emotion, userName)
} catch (e: Exception) {
    generateLocalReply(userMessage, emotion, userName)
}
```

**本地降级回复**：按情绪类型给出固定模板，确保无网络环境下系统仍可正常陪伴：

```kotlin
private fun generateLocalReply(content: String, emotion: String, userName: String?): String {

        "开心", "满意" -> "听到${name}这么说，我也很高兴！继续保持好心情哦。"
        "担心"         -> "别太担心了${name}，有什么事慢慢来，我陪着您呢。"
        "孤单"         -> "我一直在这里陪着${name}，有空也可以给孩子们打个电话聊聊。"
        "难过"         -> "我理解${name}的心情，难过的时候说出来会好一些，我在听。"

        else           -> "谢谢${name}的分享，我会一直陪着您的，有什么想聊的随时说。"
    }
}
```

**消息流订阅**：`getSessionMessages(sessionId)` 返回 `Flow<List<ConversationMessageEntity>>`，UI 层通过 `collectAsStateWithLifecycle` 订阅，数据库新增消息后自动触发 Compose 重组，无需手动刷新列表。

#### 4.4.4 情绪日志自动汇总与老人端展示

**情绪日志自动汇总**：每次会话结束时，`endSession()` 调用 `updateTodayEmotionLog()` 对当天所有会话数据进行汇总，写入或更新 `emotion_log` 表中当天的记录。`getTodayStartTimestamp()` 通过 `Calendar` 将时、分、秒、毫秒全部置零，确保同一天的多次会话始终汇总到同一条日志记录：

```kotlin
private suspend fun updateTodayEmotionLog() {
    val todayStart = getTodayStartTimestamp()
    val sessions    = db.conversationSessionDao().getByTimeRange(todayStart, todayEnd)
    val messages    = db.conversationMessageDao().getByTimeRange(todayStart, todayEnd)
    val userMessages = messages.filter { it.role == "user" }
    val emotions    = userMessages.map { it.emotion }.filter { it.isNotBlank() }

    val dominant = EmotionAnalyzer.dominantEmotion(emotions)
    val distJson = EmotionAnalyzer.emotionDistributionJson(emotions)

    val existing = db.emotionLogDao().getByDay(todayStart)
    val log = EmotionLogEntity(
        id = existing?.id ?: 0,
        dayTimestamp = todayStart,
        dominantEmotion = dominant,
        emotionDistributionJson = distJson,
        conversationCount = sessions.size,
        totalMessages = messages.size,
        summary = "今日对话${sessions.size}次，主要情绪：$dominant",
        sentToFamily = existing?.sentToFamily ?: false,
        createdAt = existing?.createdAt ?: System.currentTimeMillis()
    )
    db.emotionLogDao().insert(log)   // OnConflictStrategy.REPLACE 实现幂等写入
}
```

`EmotionLogEntity` 每天一条记录，包含主导情绪、分布 JSON、对话次数、总消息数、是否已发送给子女等字段。`EmotionLogDao` 提供按时间范围查询、按天精确查询、标记已发送等接口，`markAsSent()` 将 `sentToFamily` 字段更新为 1，避免重复发送。`endSession()` 在汇总情绪日志的同时，还将本次会话内容写入 `DiaryEntryEntity`，兼容子女端的陪伴记录展示。

**老人端情绪日志弹窗**：顶部导航栏右侧的"我的日志"按钮触发 `MyEmotionLogsDialog`，通过 `db.emotionLogDao().getAll().collectAsStateWithLifecycle()` 实时订阅情绪日志，展示最近 14 天的记录。每行显示日期（`MM月dd日`）、情绪标签（彩色背景）、对话次数三列，情绪标签颜色按语义区分：


无记录时显示引导文字"暂无记录，多和我聊聊天吧～"，鼓励用户使用。

在华为 Nova 系列真机上对 Whisper Tiny 端侧模型进行了实际测试。测试过程中发现，Android 原生 `SpeechRecognizer` 在华为设备上因 HiTouch 系统服务长期占用 `VOICE_RECOGNITION` 音频源，导致 `SpeechRecognizer` 持续返回 `ERROR_RECOGNIZER_BUSY`（错误码 8），无法正常工作。

基于此，项目完全放弃 Android 原生语音识别，改为纯 Whisper 本地推理方案。实测结果如下：

**表5.1 Whisper Tiny 端侧推理性能测试结果**

| 测试项 | 测试结果 |
|--------|----------|
| 模型初始化成功率 | 100%（冷启动约 2-3 秒） |
| 短句识别速度（5秒音频） | 约 14 秒（realtime factor ≈ 0.36x） |
| 普通话识别准确率（安静环境） | 可用，常见词汇识别正确 |
| 离线可用性 | 完全离线，无需网络 |
| 设备兼容性 | 华为 Nova 系列（arm64-v8a）验证通过 |

注：realtime factor 0.36x 表示处理 1 秒音频需约 2.8 秒，对于老年人短句输入（通常 3-8 秒）总延迟在可接受范围内。

#### 5.1.2 录音质量保障机制验证

通过 `AudioRecorder` 的最小采样数校验（`audioData.size < 1600`，即 0.1 秒），有效过滤了误触导致的极短录音。静音检测机制在实测中能够在用户停止说话约 1 秒后自动停录，减少了手动操作步骤。

## 6 总结与展望

### 6.1 技术总结

本项目基于 Android 平台，采用 Kotlin + Jetpack Compose 技术栈，实现了面向银发群体的智能膳食管理系统。核心技术贡献包括：

1. **端云协同架构**：Whisper.cpp 端侧语音转写 + PaddleOCR 端侧 OCR + 通义千问云端增强，在隐私保护与功能效果间取得平衡；
2. **分层健康风险评估引擎**：烹饪工艺 + 食材属性 + 组合效应 + 个性化慢病耦合 + 过敏原一票否决，五步评分体系实现精细化风险分级；
3. **加权情绪分析算法**：8种情绪类型、差异化权重系数、负面情绪优先感知，适配老年人情感表达特点；
4. **多轮对话会话管理**：完整的会话生命周期管理、情绪自动汇总、LLM 优先/本地降级双轨策略；
5. **图像质量评估机制**：亮度、过曝、对比度、清晰度四维评估，引导老年人拍摄高质量图像；
6. **适老化 UI 设计**：大字体、超大按钮、高对比度、语音播报、脉冲动画反馈，降低老年人使用门槛。

### 6.2 局限性与展望

1. **Whisper 识别速度**：Tiny 模型 realtime factor 约 0.36x，5秒音频需约14秒处理，对于长句输入延迟较明显。后续可探索 Small 模型量化版本或 ONNX Runtime 加速；
2. **情绪分析精度**：当前基于关键词匹配，对隐晦情绪表达（如"没什么胃口"暗示孤独）识别能力有限。后续可引入轻量级语义模型（如 MiniLM）提升语义理解能力；
3. **冰箱识别离线能力**：食材视觉识别完全依赖云端 qwen-vl-max，断网时功能不可用。后续可探索端侧轻量化视觉模型（如 MobileNet + 迁移学习）作为离线降级方案；
4. **多方言支持**：Whisper Tiny 对普通话识别效果较好，但对方言（如粤语、闽南语）支持有限，影响部分老年用户体验。
